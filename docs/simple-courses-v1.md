# Simple Courses v1

Последнее обновление: 2026-05-20

## Идея

Простой курс — это последовательность маленьких шагов, которые попадают в обычную очередь напоминаний по одному.

Курс не является отдельной LMS и не заменяет карточки. Он использует уже существующую механику:

- очередь решает, что показать пользователю сейчас;
- карточка остаётся единицей доставки;
- курс хранит порядок шагов и прогресс прохождения.

## Что входит в v1

V1 должен быть максимально простым:

- курс создаётся вручную владельцем;
- курс состоит из упорядоченных шагов;
- шаг может быть материалом, практикой или вопросом для самопроверки;
- пользователь запускает курс себе;
- первый шаг попадает в очередь;
- после просмотра шага система готовит следующий шаг;
- в карточке видно контекст: курс, номер шага, всего шагов.

V1 не включает:

- LLM-генерацию курса;
- автопроверку ответов;
- публичный маркетплейс курсов;
- роли авторов и покупателей;
- сложные версии курса;
- отдельный полноэкранный LMS-интерфейс.

## UX

Пользовательский сценарий:

1. Пользователь запускает курс.
2. В очереди появляется первый шаг.
3. Пользователь вытягивает карточку курса.
4. После просмотра карточка считается пройденным шагом.
5. Следующий шаг планируется в очередь.
6. Если пользователь хочет учиться быстрее, он может вытягивать следующие шаги заранее через ту же очередь.

Карточка курса должна выглядеть как обычная карточка очереди, но с небольшим контекстом:

```text
Курс: Основы SQL
Шаг 3 из 12

JOIN соединяет строки из двух таблиц...
```

## Типы шагов

`material`

Материал для чтения, просмотра или осмысления.

`practice`

Маленькое действие, которое нужно сделать.

`question`

Вопрос для самопроверки. В v1 без автопроверки: ответ может быть спрятан в тексте шага или раскрыт в будущей версии интерфейса.

## Модель данных

Нужны отдельные сущности курса, потому что карточка — это доставка, а не структура программы.

Минимальные таблицы:

- `courses` — шаблон курса;
- `course_steps` — упорядоченные шаги курса;
- `course_enrollments` — прохождение курса конкретным пользователем или очередью;
- `course_step_deliveries` — связь между шагом курса и карточкой/напоминанием, которое было создано для очереди.

Важный принцип: карточка, созданная из шага курса, остаётся обычной карточкой. Связь с курсом хранится отдельно.

## Scheduling

V1 поддерживает два режима выдачи:

`after_view`

Следующий шаг появляется после просмотра текущего. Это самый простой и интерактивный режим.

`daily`

Один шаг в день. Используется для спокойного курса, который сам возвращается в очередь.

По умолчанию для первого релиза лучше начать с `after_view`: он проще, хорошо ложится на pull-queue и не требует отдельной настройки времени.

## Интеграция с очередью

Курс должен добавлять шаги в ту же очередь, что и обычные карточки:

- личный курс использует `queue_scope_type = user`;
- будущий групповой курс может использовать `queue_scope_type = chat`;
- Mini App queue не должна знать, что карточка пришла из курса, кроме отображения метаданных;
- действия очереди остаются знакомыми: просмотрено, не просмотрено, перенести, архив.

После `viewed` для карточки курса система должна:

1. отметить delivery как пройденный;
2. сдвинуть enrollment на следующий шаг;
3. если следующий шаг есть, создать для него следующую карточку/напоминание;
4. если шагов больше нет, завершить enrollment.

## Acceptance Criteria

- Можно создать курс из нескольких текстовых шагов.
- Можно запустить курс для пользователя.
- Первый шаг появляется в личной очереди.
- После просмотра шага следующий шаг появляется в очереди.
- Прогресс курса сохраняется отдельно от карточек.
- Обычные карточки, one-time reminders и chat queues не меняют поведение.

## Backend foundation

Первый backend foundation:

1. Добавлены таблицы `courses`, `course_steps`, `course_enrollments`, `course_step_deliveries`.
2. Добавлен store-контракт для создания курса, добавления шагов, запуска курса и выпуска следующего шага.
3. `viewed` в очереди умеет завершать course-step delivery и выпускать следующий шаг.
4. Пока не сделан Mini App UI; курс можно подключить следующим срезом через owner-only API или экран управления.

Следующий срез:

- owner-only API для создания курса и шагов;
- Mini App экран списка курсов;
- кнопка запуска курса;
- отображение прогресса курса в карточке очереди более компактным UI, а не только текстом внутри карточки.
