# Бесплатный хостинг для Telegram-бота: сравнительный обзор

| Платформа | Тип запуска | Преимущества | Ограничения | Релевантность для бота |
| --- | --- | --- | --- | --- |
| **Railway free tier** | Контейнер/сервер | • Простой деплой из GitHub<br>• Постоянно запущенный контейнер (до лимита, ~500 GB·часов/мес)<br>• Cron/график задач и переменные окружения | • Засыпает, если превышен лимит ресурсов<br>• Нужна привязка карты<br>• Нет бесплатного выделенного IP | Подходит для long polling/webhook + фоновых задач; нужно следить за лимитами времени и диска. |
| **Render Free Web Service + Background Worker** | Контейнер/сервер | • Понятный интерфейс, авто-деплой из Git<br>• Бесплатные веб‑сервисы и воркеры (750 CPU-часов/мес)<br>• Логирование в панели | • Спящие сервисы при простое, прогрев до 30 сек<br>• Ограничение на выходящие запросы при простое<br>• SSH-доступ только на платном плане | Хорош для webhook (если принять прогрев) и фонового планировщика; long polling возможен, но downtime при sleep. |
| **Fly.io Free Allowance** | Контейнер/VM ближе к bare metal | • Geo-распределение, любой язык<br>• 3 shared-cpu-1x VMs + 3 GB RAM-часов бесплатно<br>• Можно поднять постоянный long polling | • Конфигурация через CLI, чуть сложнее вход<br>• Лимиты трафика и хранения, требуется кредитка | Позволяет держать постоянно включённый бот с доступом к локальному диску/volume; больше контроля, но сложнее настройка. |
| **Cloudflare Workers + KV/D1** | Serverless функции | • Огромный бесплатный лимит запросов<br>• Глобальная сеть, высокая доступность<br>• Cron Triggers (ежеминутно) | • Нет постоянного процесса: long polling невозможен<br>• Ограничение на время выполнения (⩽30 сек Workers, ⩽15 мин в Queues)<br>• Хранение state требует KV/D1 | Подходит для webhook-варианта: Telegram сам пушит обновления. Нужно внешний keep-alive для фоновых повторений или Cloudflare Queues/Workers Cron. |
| **Vercel Serverless Functions + Cron Jobs** | Serverless функции | • Автодеплой Next.js/Node проектов<br>• Cron Jobs (Beta) для периодических задач<br>• Хорошая интеграция с GitHub Secrets | • Нет постоянных процессов, long polling не поддержан<br>• Ограничение 10 сек (Hobby) на выполнение<br>• Вебхуки спят, если нет трафика; динамический IP | Подходит для webhook модели, если бот всегда получает запросы от Telegram. Для генерации интервалов нужны Cron Jobs или внешние очереди. |
| **Deta Space** | Serverless/микросервисы | • Полностью бесплатный план, простое API<br>• Персистентное хранение (Deta Base/Drive)<br>• Фоновый Scheduler | • Ограничение по памяти (512 МБ) и CPU<br>• Vendor lock-in API<br>• Меньшая экосистема по сравнению с Railway/Render | Можно держать webhook обработчик и scheduler для повторений; минимальные расходы. |
| **Supabase Edge Functions + Scheduled Triggers** | Serverless + Postgres | • Бесплатный Postgres + Auth + Edge Functions<br>• Cron-триггеры<br>• Удобное хранение данных и API | • Edge Functions ограничены 10 сек на выполнение<br>• Постоянные процессы невозможны<br>• Бэкенд на Postgres → нужно следить за квотами | Подойдёт, если хочется сразу Postgres и webhook. Для длинных фоновых задач придётся использовать `pg_cron` или внешние воркеры. |

## Выводы
- **Нужен постоянный процесс / long polling** → Railway, Fly.io или Render Worker — позволяют держать всегда включённый бот.
- **Webhook + serverless** → Vercel, Cloudflare Workers, Supabase или Deta Space; нужно убедиться, что Telegram может достучаться до URL и что интервальные напоминания можно запускать Cron-триггером.
- **Хранилище**: Railway/Render/Fly позволяют использовать любую БД (в т. ч. локальную). Cloudflare/Supabase/Deta предлагают встроенные KV/Base/Postgres.
- **Настройка деплоя и секретов**: все сервисы поддерживают GitHub интеграцию и переменные окружения; для локальной разработки всё равно понадобится `.env`.
